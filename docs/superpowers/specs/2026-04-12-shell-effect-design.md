# Shell Effect

Add a `:shell` effect for spawning subprocesses with stdin/stdout/stderr capture, following the same async pattern as `:http`.

## Fennel API

```fennel
;; In an event handler, return a :shell effect
{:db new-db
 :shell {:cmd ["bb" "smar.bb.clj" "complete"]
         :stdin (redin.json_encode request-payload)
         :on-success :event/process-done
         :on-error :event/process-failed}}
```

Response delivered as an event:

```fennel
;; on-success (exit code 0)
[:event/process-done {:stdout "..." :stderr "..." :exit-code 0}]

;; on-error (non-zero exit code or spawn failure)
[:event/process-failed {:stdout "..." :stderr "..." :exit-code 1 :error "..."}]
```

## Odin side

### New file: `src/host/bridge/shell.odin`

Mirrors `http_client.odin`:

```odin
Shell_Request :: struct {
    id:    string,
    cmd:   []string,
    stdin: string,
}

Shell_Response :: struct {
    id:        string,
    stdout:    string,
    stderr:    string,
    exit_code: int,
    error_msg: string,
}

Shell_Client :: struct {
    results:       [dynamic]Shell_Response,
    results_mutex: sync.Mutex,
}
```

- `shell_client_request` spawns a thread that:
  1. Creates pipes for stdin/stdout/stderr via `os.pipe()`
  2. Calls `os.process_start` with `Process_Desc{command, stdin, stdout, stderr}`
  3. Writes `request.stdin` to the stdin pipe, closes it
  4. Reads stdout and stderr pipes to completion
  5. Calls `os.process_wait` for exit code
  6. Pushes `Shell_Response` into mutex-protected results queue
- `shell_client_poll` drains results (called each frame from main loop)

### Bridge changes in `bridge.odin`

New host function:

```odin
redin_shell :: proc "c" (L: ^Lua_State) -> i32
```

Reads from Lua stack: `(id: string, cmd: table, stdin: string)`

- `id` â€” unique request identifier (generated by the effect executor)
- `cmd` â€” sequential Lua table of strings (command + args)
- `stdin` â€” string to write to process stdin (empty string if none)

Registered as `redin.shell` in the `init` proc.

### Bridge: response delivery

`deliver_shell_response` in `bridge.odin`, same pattern as `deliver_http_response`:

```
redin_events([[:shell-response {id, stdout, stderr, exit-code, error}]])
```

Called from `poll_shell` which is called each frame in `main.odin`.

### Main loop change

In `main.odin`, add `bridge.poll_shell(&b)` alongside `bridge.poll_http(&b)`.

### Bridge struct change

Add `shell_client: Shell_Client` to the `Bridge` struct. Init and destroy in `bridge.init` / `bridge.destroy`.

## Fennel side

### Changes to `effect.fnl`

Add `:shell` effect executor, mirroring `:http`:

```fennel
(M.reg-fx :shell
  (fn [params]
    (set next-shell-id (+ next-shell-id 1))
    (let [id (tostring next-shell-id)
          cmd (or params.cmd [])
          stdin (or params.stdin "")]
      (tset pending-shell id {:on-success params.on-success
                              :on-error params.on-error})
      (when _G.redin_shell
        (_G.redin_shell id cmd stdin)))))
```

Add `handle-shell-response` to route responses:

```fennel
(fn M.handle-shell-response [response]
  (let [req-info (. pending-shell response.id)]
    (when req-info
      (tset pending-shell response.id nil)
      (let [dispatch-fn (or _G.dispatch _G.redin_dispatch)]
        (when dispatch-fn
          (if (= response.exit-code 0)
            (dispatch-fn [req-info.on-success response])
            (dispatch-fn [req-info.on-error response])))))))
```

New module-level state: `(var pending-shell {})` and `(var next-shell-id 0)`.

### Changes to `view.fnl`

Add `:shell-response` case in `deliver-events`, same as `:http-response`:

```fennel
(= event-type :shell-response)
(let [effect-mod (require :effect)]
  (effect-mod.handle-shell-response (. event 2)))
```

### Bridge global

Expose `redin_shell` as a flat global (same as `redin_http`):

```odin
lua_pushcfunction(b.L, redin_shell)
lua_setglobal(b.L, "redin_shell")
```
